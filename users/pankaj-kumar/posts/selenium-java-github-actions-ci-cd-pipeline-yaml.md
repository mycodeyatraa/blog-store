# Automating the Automation: Building a CI Pipeline with GitHub Actions

In our last tutorial, we containerized our Selenium Grid using Docker. Our framework is now decoupled from our local operating system.

But we still have a manual bottleneck: A human engineer has to sit at their laptop, open a terminal, and type `mvn clean test` to trigger the execution.

If a developer pushes broken code to the `main` branch at 2:00 AM, our Selenium framework won't catch it until the QA Engineer comes into the office at 9:00 AM. 

This is where **Continuous Integration (CI)** comes in. We need a server that listens for code changes and automatically triggers our Selenium tests. Today, we will build this using **GitHub Actions**.

---

## 1. What is GitHub Actions?

GitHub Actions is a CI/CD runner built directly into GitHub. It allows you to automate workflows directly inside your repository.

When a developer creates a Pull Request or pushes code to the `main` branch, GitHub will automatically spin up a temporary, disposable Ubuntu Linux Virtual Machine, checkout your code, install Java, run your Maven tests, and post the results—all in the cloud.

---

## 2. Creating the YAML Workflow

GitHub Actions are defined using YAML files placed in a very specific directory: `.github/workflows/`.

Create a file named `.github/workflows/selenium-tests.yml` in your repository:

```yaml
name: Nightly Selenium Regression
# 1. When should this pipeline run?
on:
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]
  schedule:
    # Run automatically at 2:00 AM every night
    - cron: '0 2 * * *'
# 2. What should it do?
jobs:
  run-ui-tests:
    # 3. Where should it run? (A disposable Ubuntu VM)
    runs-on: ubuntu-latest
    steps:
    # Step A: Checkout the code from GitHub onto the VM
    - name: Checkout Code
      uses: actions/checkout@v3
    # Step B: Install Java 17 on the VM
    - name: Set up JDK 17
      uses: actions/setup-java@v3
      with:
        java-version: '17'
        distribution: 'temurin'
        cache: maven # Speeds up future builds
    # Step C: Spin up our Docker Selenium Grid!
    - name: Start Docker Selenium Grid
      run: docker-compose up -d
    # Step D: Wait for the Grid to be fully healthy
    - name: Wait for Grid
      run: sleep 15
    # Step E: Run the TestNG Suite using Maven
    - name: Execute Maven Tests
      run: mvn clean test -DsuiteXmlFile=testng.xml
    # Step F: Upload the Allure Results as an Artifact
    - name: Upload Test Results
      if: always() # Run this even if the tests fail!
      uses: actions/upload-artifact@v3
      with:
        name: allure-results
        path: target/allure-results
        retention-days: 7
```

---

## 3. Breaking Down the Pipeline

Let's look at exactly what happens when you push code:

1. **The Trigger (`on: push`):** GitHub detects your code push and instantly provisions a brand new Ubuntu Linux VM.
2. **Environment Setup:** It runs standard GitHub Actions modules (`uses: actions/...`) to pull your repository code into the VM and install the Java Development Kit (JDK).
3. **Infrastructure as Code:** Because the VM has Docker pre-installed, it executes `docker-compose up -d`. This instantly spins up the isolated Chrome/Firefox Selenium Grid we built in the previous tutorial!
4. **Execution:** It runs `mvn clean test`. Your Java code executes, sending WebDriver commands to the Docker Grid running right next to it on the VM.
5. **Artifacts:** Regardless of whether the tests pass or fail (`if: always()`), it zips up the raw Allure JSON files generated by TestNG and attaches them to the GitHub Actions dashboard for the team to download!

## 4. Securing Passwords and API Keys

If your tests log into a database, you probably have a password hardcoded in your `application.properties` file. **Never push passwords to GitHub.**

Instead, use **GitHub Secrets**.

1. Go to your GitHub Repository -> Settings -> Secrets and Variables -> Actions.
2. Create a new Secret named `DB_PASSWORD`.
3. Update your YAML file to inject the secret into the VM's Environment Variables:

```yaml
    - name: Execute Maven Tests
      env:
        DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
      run: mvn clean test -DsuiteXmlFile=testng.xml
```

Now, your Java code can securely read the password using `System.getenv("DB_PASSWORD")` without exposing it in your source code!

## Conclusion

By writing a 40-line YAML file, you have achieved true **Continuous Integration**. 

Your automated tests no longer require human intervention. They run automatically on every code push, and they run on a scheduled cron job every single night. The framework spins up its own disposable Docker infrastructure, executes the tests, and tears everything down when it finishes.
