# Event-Driven Automation: Validating Kafka Streams in Selenium

In our previous tutorials, we validated data at rest. We queried MySQL tables, searched MongoDB documents, and parsed GraphQL APIs. 

But what if the data isn't resting? What if it's streaming?

Modern enterprise architectures are rapidly shifting toward **Event-Driven Architectures** powered by **Apache Kafka**. If a user clicks "Submit Order" on the UI, the backend might not immediately write to a database. Instead, it might publish an "OrderPlaced" event to a Kafka Topic.

If your Selenium test simply waits for a UI success banner, it is completely blind to the Kafka stream. In this tutorial, we will learn how to build a Kafka Consumer inside our Selenium Java framework to intercept and validate real-time event streams!

---

## 1. What is Apache Kafka?

Kafka is a distributed event streaming platform. Instead of rows or documents, Kafka deals in **Events** (or "Messages").

- **Producers**: Applications that publish events (e.g., your E-Commerce backend).
- **Topics**: Categories where events are published (e.g., `orders-topic`).
- **Consumers**: Applications that subscribe to topics and read the events.

In our End-to-End test, Selenium will act as a **Consumer**.

---

## 2. Adding the Kafka Java Client

To allow our automation framework to consume messages from a Kafka cluster, we need the official `kafka-clients` library in our `pom.xml`.

```xml
<dependency>
    <groupId>org.apache.kafka</groupId>
    <artifactId>kafka-clients</artifactId>
    <version>3.6.1</version>
</dependency>
```

We will also need Google's **Gson** library (which we used in our API tutorials) to easily parse the JSON messages we receive from Kafka.

---

## 3. Writing the Kafka Consumer Test

Let's write a Selenium test that places an order on the UI. Behind the scenes, the application publishes a JSON payload to the `ecommerce.orders` Kafka topic.

Our test will connect to the Kafka broker, poll the topic for the next 10 seconds, and assert that our specific Order ID was successfully broadcasted to the stream!

```java
import org.apache.kafka.clients.consumer.ConsumerConfig;
import org.apache.kafka.clients.consumer.ConsumerRecord;
import org.apache.kafka.clients.consumer.ConsumerRecords;
import org.apache.kafka.clients.consumer.KafkaConsumer;
import org.apache.kafka.common.serialization.StringDeserializer;
import org.openqa.selenium.By;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import org.testng.Assert;
import org.testng.annotations.AfterMethod;
import org.testng.annotations.BeforeMethod;
import org.testng.annotations.Test;
import java.time.Duration;
import java.util.Collections;
import java.util.Properties;
public class KafkaEventValidationTest {
    WebDriver driver;
    // Kafka Broker Configuration
    String bootstrapServers = "localhost:9092";
    String topicName = "ecommerce.orders";
    @BeforeMethod
    public void setup() {
        driver = new ChromeDriver();
    }
    @Test
    public void testOrderEventPublishedToKafka() {
        // --- PART 1: The UI Workflow ---
        driver.get("https://practice.mycodeyatra.com/checkout");
        driver.findElement(By.id("product-id")).sendKeys("PROD-99");
        driver.findElement(By.id("place-order-btn")).click();
        // Extract the Order ID generated by the UI
        String uiOrderId = driver.findElement(By.id("order-id-label")).getText();
        Assert.assertNotNull(uiOrderId);
        // --- PART 2: The Kafka Validation ---
        // 1. Configure the Kafka Consumer Properties
        Properties props = new Properties();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, bootstrapServers);
        // Every consumer needs a group ID. We generate a random one to ensure we read from the latest offset.
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "selenium-test-group-" + System.currentTimeMillis());
        // We expect the Keys and Values in the Kafka Topic to be Strings
        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());
        // Start reading from the very latest messages published to the topic
        props.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, "latest");
        boolean messageFound = false;
        // 2. Initialize the Kafka Consumer
        try (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)) {
            // 3. Subscribe to the specific Topic
            consumer.subscribe(Collections.singletonList(topicName));
            System.out.println("Polling Kafka Topic: " + topicName + " for Order: " + uiOrderId);
            // 4. Poll the topic for messages! 
            // Because Kafka is asynchronous, the message might take a few seconds to arrive.
            // We loop and poll for a maximum of 10 seconds.
            long endTime = System.currentTimeMillis() + 10000;
            while (System.currentTimeMillis() < endTime) {
                // Poll for new records, wait up to 1 second per poll
                ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(1000));
                for (ConsumerRecord<String, String> record : records) {
                    String jsonPayload = record.value();
                    System.out.println("Received Event: " + jsonPayload);
                    // 5. Assert: Does the event payload contain our UI Order ID?
                    if (jsonPayload.contains("\"orderId\":\"" + uiOrderId + "\"")) {
                        messageFound = true;
                        // We could use Gson here to parse the JSON and assert specific fields!
                        Assert.assertTrue(jsonPayload.contains("\"status\":\"PENDING\""), 
                            "Order status should be PENDING in the Kafka stream!");
                        break; // Exit the loop, we found our message!
                    }
                }
                if (messageFound) break;
            }
        }
        // 6. Final Assertion
        Assert.assertTrue(messageFound, "CRITICAL BUG: The Order event never arrived in the Kafka Topic!");
    }
    @AfterMethod
    public void teardown() {
        driver.quit();
    }
}
```

---

## 4. The Complexity of Asynchronous Testing

The most critical part of the code above is the `while` loop.

When you execute a JDBC query, the database responds immediately. But Kafka is inherently asynchronous. When you click "Submit Order" on the UI, it might take 200 milliseconds, or it might take 5 seconds for the backend microservice to finally publish the event to Kafka.

If you just `poll()` exactly once and assert immediately, your test will randomly fail with a "False Negative" because the message simply hadn't arrived yet. 

You **must** implement a polling loop with a timeout (e.g., 10 seconds) to continuously check the topic. This is the Kafka equivalent of a Selenium `WebDriverWait`!

## Conclusion

Event-Driven Architecture is completely invisible to the frontend browser. A user has no idea if an event was published to Kafka or not.

By building a dynamic Kafka Consumer directly into your Selenium tests, you verify that the UI action successfully triggered the backend microservice to broadcast the correct data payload to the rest of the enterprise ecosystem!
